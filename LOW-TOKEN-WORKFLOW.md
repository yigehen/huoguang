# 《火光深处等春来》低 Token 工作模式

用于后续续写、修文、同步项目资料时，尽量减少无效上下文和空转轮次。

## 一、默认读取顺序（最小必要上下文）
每次继续小说前，优先只读这些文件：

1. `chapters/chapter-index.md`
2. `notes/project-notes.md`
3. `notes/mystery-thread.md`
4. `notes/relationship-map.md`
5. `notes/timeline.md`
6. 最新一章正文（必要时再加前一章）

除非用户明确要求回溯更早内容，否则不要一口气重读大量旧章节。

## 二、写作执行规则
- 用户一旦明确要求“继续写”，直接落正文文件，不先长篇规划。
- 优先按批次写完目标章节，再统一同步配套文件。
- 每章必须有：
  - 章名
  - 正文
  - 明确收尾

## 三、进度汇报规则
只允许汇报**真实文件级进度**。

可汇报内容：
- 新建了哪个章节文件
- 修订了哪个章节文件
- 更新了哪些配套文件
- 是否已 commit / push

禁止汇报内容：
- “我在想结构”
- “方向已经稳了”
- “我正在推进”
- 任何没有落盘证据的内部进展

## 四、配套同步最小集合
每次小说写作批次完成后，至少同步：
- `chapters/chapter-index.md`
- `notes/project-notes.md`
- `notes/mystery-thread.md`
- `notes/relationship-map.md`
- `notes/timeline.md`
- `README.md`

## 五、减少上下文浪费的原则
- 能从项目文件读取的信息，不从长聊天历史回忆。
- 能读最新章和项目 notes 解决的问题，不重读大量旧章。
- 能一次写完再统一同步，就不要多轮“写一点、报一点、再想一点”。
- 能在文件里固定的长期状态（写到第几章、最新提交、下一阶段方向），就写进项目文件和 MEMORY，而不是依赖会话上下文。

## 六、默认结果汇报格式
采用“结论 + 简短进度”：

- 结论：本轮是否完成
- 简短进度：列出新增章节文件、更新的配套文件、是否已推 GitHub

## 七、下一步触发词
当用户说这些词时，直接进入本模式：
- “继续小说”
- “继续写”
- “写 56-60”
- “修 41-45”
- “切回小说”

执行时不要先重复规则，直接开始。
